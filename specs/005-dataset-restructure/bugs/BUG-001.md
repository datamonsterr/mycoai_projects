# Bug Report: Metadata Schema and Source Parsing Do Not Match Actual Dataset Structure

**Type**: Spec gap + Implementation drift
**Severity**: Critical
**Feature**: 005-dataset-restructure
**Reported**: 2026-05-07
**Status**: Patched
**Patched**: 2026-05-07

## Description

The current implementation (dataset.py, config.py, data-model.md) makes incorrect assumptions about the real on-disk structure of both source collections. The `Dataset/new_data/` directory is organized as letter ranges (`D - L/`, `M - R/`, `S - Z/`) that the pipeline must skip — only the inner `{species}/{strain (Txxx)}/` hierarchy matters. Within each leaf folder, images encode environment in the filename (e.g. `T(N) CYA rev.JPG`) and ob/rev are angle labels that should become leaf subdirectories (`ob/`, `rev/`). The metadata schema must change from per-image `item.json` files to one consolidated JSON array with a redesigned schema matching consumer needs (upload_qdrant.py requires segments, not full images).

### Specific Gaps

1. **Letter-range folders ignored**: Code walks `Dataset/new_data/` flatly; actual structure has `D - L/`, `M - R/`, `S - Z/` grouping folders that must be transparently entered
2. **Angle (ob/rev) becomes directory**: Leaf folders must be `ob/` and `rev/` per strain subfolder, not part of filename
3. **Segments per leaf directory**: Each ob/rev leaf folder must contain `segments/` with `segment_1.{ext}` files; path stored in metadata
4. **Per-item item.json removed**: Replace with single consolidated JSON array at collection root
5. **Metadata schema redesign**:
   - `item_id` — UUID5 from source path for stable cross-run identity (needed so Qdrant/fine-tune/retrieval consumers can re-look up same item after re-preparation)
   - `paths` — object (not `_path` suffixes): `{source, prepared, segments, bbox_kmeans, bbox_contour, pipeline_kmeans, pipeline_contour}`
   - `instance_info` — `{species, strain, environment, angle}` (flat, not nested `data`)
   - `segmentation` — `{kmeans: [{x, y, w, h}, ...], contour: [{x, y, w, h}, ...]}` per method
6. **Source rename atomically**: `Dataset/original/` → `Dataset/curated_primary/`, `Dataset/new_data/` → `Dataset/incoming_low_quality/` as mv on disk

## Root Cause Analysis

The original spec (written from user's initial description "I want to work with dataset...") did not explicitly enumerate the physical structure of `Dataset/new_data/`. The user's intent was captured at a high level ("reformat both dataset") but the specific nesting (letter ranges → species → Txxx strain → images with environment in filename) was never inspected before phase 2 implementation. The data-model.md and dataset.py were written against an idealized flat source structure that doesn't exist. Additionally, the metadata schema was designed to mirror the old flat `segmented_image/` consumer pattern rather than the new consolidated, path-object-based consumption pattern now requested.
