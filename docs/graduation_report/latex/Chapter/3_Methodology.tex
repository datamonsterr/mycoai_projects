\section{Requirements Analysis}\label{requirements-analysis}

The web application translates the MycoAI Retrieval model into a governed product for two actor classes: \textbf{User} and \textbf{Data Owner}. Requirements are defined in \texttt{docs/SRS.md} and refined through the technical and feature specifications in the product repositories.

\subsection{Actor Model and Permissions}\label{actor-model-and-permissions}

\begin{center}
\begin{tabular}{@{}p{3cm} p{4.5cm} p{6.5cm}@{}}
\toprule
\textbf{Actor} & \textbf{Role Value} & \textbf{Capabilities} \\
\midrule
\textbf{User} & \texttt{user} & Upload images, retrieve species predictions, review/edit bounding boxes, download batch results, submit feedback and contribution proposals \\
\textbf{Data Owner} & \texttt{owner} & All User capabilities plus index reference data, CRUD species/media metadata, browse dataset, review feedback, re-index \gls{qdrant}, assess candidate models, and manage users \\
\bottomrule
\end{tabular}
\end{center}

The Data Owner role inherits all User permissions. Backend RBAC is enforced through FastAPI dependencies, while the frontend conditionally renders owner-only pages and actions.

\subsection{Key Design Decisions}\label{key-design-decisions}

\textbf{UUID primary keys:} all entity tables use UUIDs to avoid sequential enumeration and simplify distributed-safe creation.

\textbf{Soft delete pattern:} species, \gls{media}, strain, image, and segment records use archive flags instead of permanent deletion.

\textbf{JSONB for flexible state:} retrieval jobs, training progress, and audit payloads use JSONB fields where rigid relational columns would be unnecessarily restrictive.

\section{Core Backend Pipelines}\label{core-backend-pipelines}

\subsection{Segmentation Pipeline}\label{segmentation-pipeline}

Segmentation is implemented in the backend as a KMeans-driven preprocessing and bounding-box extraction pipeline. The deployed application uses this deterministic path, while the research chapter evaluates YOLO segmentation as a stronger alternative for flare-heavy edge cases.

The backend artifact layout per image is:

\begin{verbatim}
uploads/{strain}/{media}/{image_id}/
+-- source.jpg
+-- prepared.jpg
+-- bbox_kmeans.jpg
+-- pipeline_kmeans.jpg
\-- segments/
    +-- segment_0.jpg
    +-- segment_1.jpg
    \-- segment_2.jpg
\end{verbatim}

\subsection{Retrieval Pipeline}\label{retrieval-pipeline}

The retrieval orchestrator performs feature extraction, Qdrant nearest-neighbor search, aggregation across segments, and final ranking. Same-media retrieval is used when the query medium is managed; otherwise the system falls back to all-media search.

\subsection{Qdrant Vector Database Design}\label{qdrant-vector-database-design}

The Qdrant collection stores per-segment points with multiple named vectors so different feature extractors can be queried or compared without duplicating payload metadata.

\subsection{Model and Index Maintenance}\label{model-and-index-maintenance}

The Model Index page manages re-indexing and candidate-model assessment, while heavy retraining remains an offline workflow tied to the research pipeline.

\subsection{CI/CD Pipeline}\label{cicd-pipeline}

GitHub Actions run on every push and pull request:

\begin{itemize}
\item \textbf{Backend}: ruff check, format check, mypy (strict), then pytest.
\item \textbf{Frontend}: ESLint, TypeScript typecheck, then Vite build.
\item \textbf{Docker}: image build dry-run verification.
\end{itemize}

Deployment to production is manual:

\begin{itemize}
\item \texttt{docker\ compose\ pull}
\item \texttt{docker\ compose\ up\ -d}
\item Triggered only after pull request approval and merge to \texttt{main}.
\end{itemize}

\section{Summary}\label{summary}

The implemented product combines governed metadata management, deterministic backend segmentation, vector retrieval through \gls{qdrant}, and role-based workflows for laboratory usage. Together with the research pipeline in Chapter 2, it forms a full path from experimental retrieval model to operator-facing application.
