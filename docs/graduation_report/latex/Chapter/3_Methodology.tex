\subsection{Requirements Analysis}\label{requirements-analysis}

The web application translates the MycoAI Retrieval model into a governed, multi-role product supporting two actor classes: \textbf{User} (retrieval operator) and \textbf{Data Owner} (governance operator). Requirements are defined in \texttt{docs/SRS.md} and elaborated across 13 technical design documents in \texttt{docs/technical\_spec/} and 8 feature specifications in \texttt{docs/feature\_spec/}.

\includegraphics[width=\textwidth,keepaspectratio]{figures/ch03_erd.png}\\
\emph{Figure 3.4: Entity-Relationship Diagram (ERD) for the MycoAI Retrieval PostgreSQL schema --- 14 tables with primary/foreign key relationships.}

\subsubsection{Key Design Decisions}\label{key-design-decisions}

\textbf{UUIDv4 Primary Keys:} All entity tables use UUIDv4 primary keys. This avoids sequential enumeration, supports distributed-safe key generation, and eliminates collision risk when merging datasets across instances.

\textbf{Soft Delete Pattern:} Species, Media, Strain, Image, and Segment all use an \texttt{is\_archived} boolean flag with an \texttt{archived\_at} timestamp rather than permanent deletion. Images additionally carry a \texttt{data\_update\_status} field (\texttt{current}, \texttt{updated\_requires\_reindex}, or \texttt{archived}) that drives the Qdrant re-indexing workflow.

\textbf{JSONB for Flexible Configurations:} Retrieval job configurations, training job progress, audit log change diffs, and system state values use PostgreSQL JSONB columns, balancing schema flexibility with relational integrity for core entities.

\textbf{Composite Uniqueness:} The \texttt{strains} table enforces \texttt{UNIQUE(name,\ species\_id)} --- the same strain name may exist across different species, reflecting biological reality. The \texttt{segments} table enforces \texttt{UNIQUE(image\_id,\ segment\_index)}.

\subsection{Core Backend Pipelines}\label{core-backend-pipelines}

\subsubsection{Segmentation Pipeline}\label{segmentation-pipeline}

Segmentation is implemented at \texttt{backend/src/mycoai\_retrieval\_backend/segmentation.py} (399 lines) as a \texttt{SegmentationPipeline} class with inline OpenCV and scikit-learn calls. Two methods are available:

\textbf{KMeans Method:} (1) Convert to HSV color space, (2) K=3 KMeans clustering on HSV pixels, (3) Select foreground label, (4) Spatial K=2/3 clustering for individual colonies, (5) Bounding box refinement (erosion, contour fitting, halo shrink).

\textbf{Contour Method:} (1) Canny edge detection, (2) Morphological close, (3) Find contours with circularity score filtering, (4) Select top-3 by score.

The pipeline produces the following artifact layout per image:

\begin{verbatim}
uploads/{strain}/{media}/{image_id}/
├── source.jpg              # Original uploaded image
├── prepared.jpg            # 256×256 preprocessed (plate mask applied)
├── bbox_kmeans.jpg         # Bounding box overlay visualization
├── pipeline_kmeans.jpg     # 3-panel visualization (source | prep | bbox)
└── segments/
    ├── segment_0.jpg       # Cropped colony segment
    ├── segment_1.jpg
    └── segment_2.jpg
\end{verbatim}

\subsubsection{Retrieval Pipeline}\label{retrieval-pipeline}

The retrieval orchestrator implements the full pipeline: (1) Feature Extraction for each segment crop using EfficientNetB1\_finetuned (1280-dim), (2) Qdrant KNN Search with cosine distance, configurable k (1--20), and environment filter, (3) Aggregation across all segments/images of the same strain, (4) Ranking by aggregated score.

\subsection{Summary}\label{summary}

The MycoAI Retrieval web application comprises 11 frontend pages, 47 REST API endpoints, 14 database tables, a 13-vector Qdrant collection, and a 6-service Docker Compose deployment. The system enforces RBAC across two actor roles and supports the complete lifecycle: upload → segment → retrieve → provide feedback → review → index → maintain model. Every Data Owner mutation is audit-logged, no permanent deletion is permitted, and the dual-database architecture (PostgreSQL for governance, Qdrant for search) balances consistency with performance.

The implementation is grounded in 103 lines of use case design documentation, 275 lines of database design, 371 lines of API contract, and 482 lines of SQLAlchemy ORM models --- all referenced against the SRS and verified by the CI pipeline.
